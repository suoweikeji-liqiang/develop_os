   - id: String @id @default(cuid())
   - requirementId: String (FK to Requirement)
   - version: Int
   - model: Json (NOT nullable — snapshots always have a model)
   - confidence: Json?
   - changeSource: String @default("manual") — values: "manual" | "ai-structure" | "ai-converse" | "assumption"
   - createdBy: String
   - createdAt: DateTime @default(now())
   - No updatedAt — snapshots are immutable
   - relation: requirement Requirement @relation(fields: [requirementId], references: [id], onDelete: Cascade)

   - @@index([requirementId, createdAt])

This ensures pre-Phase-4 requirements have version history. The script can be inline in the tsx invocation or a small file at scripts/backfill-versions.ts.

<task type="auto">
  <name>Task 2: Snapshot-on-write in updateModel and version tRPC router</name>
  <files>src/server/trpc/routers/requirement.ts, src/server/trpc/routers/version.ts, src/server/trpc/router.ts</files>
  <action>
1. Modify requirement.ts updateModel mutation:
   - Add `changeSource` to input schema: z.enum(['manual', 'ai-structure', 'ai-converse', 'assumption']).default('manual')
   - Wrap the entire mutation body in `prisma.$transaction(async (tx) => { ... })`
   - Before updating, read current state: `tx.requirement.findUniqueOrThrow({ where: { id: input.id }, select: { model: true, version: true, confidence: true } })`
   - If current.model is not null, create snapshot: `tx.requirementVersion.create({ data: { requirementId: input.id, version: current.version, model: current.model, confidence: current.confidence ?? undefined, changeSource: input.changeSource, createdBy: ctx.session.userId } })`
   - Then update requirement with new model and increment version (same as current logic but using `tx` instead of `prisma`)
   - Emit 'requirement.version.created' event after transaction completes (outside the transaction callback, using the returned requirement)
   - Keep existing 'requirement.updated' event emission

2. Create src/server/trpc/routers/version.ts with versionRouter:
   - `list` query: input { requirementId: string }, returns requirementVersion.findMany with SELECT only metadata (id, version, changeSource, createdBy, createdAt) — NOT the full model JSON. Order by version desc.
   - `getTwo` query: input { requirementId: string, versionA: number, versionB: number }, fetches two versions by compound unique key (requirementId_version), returns { versionA, versionB } with full model JSON. Use Promise.all for parallel fetch. Also fetch the current requirement (for the "latest" version that lives on the Requirement row itself). If versionA or versionB equals the current requirement.version, return the requirement's model instead of looking up RequirementVersion (since the latest version is not yet snapshotted).

3. Register versionRouter in src/server/trpc/router.ts:
   - Import versionRouter from './routers/version'
   - Add `version: versionRouter` to createTRPCRouter call
  </action>

  </verify>
