# Plan 03-02: Conversation API Route and tRPC Router

## Goal

Wire the server-side conversation infrastructure: streaming API route that calls `streamText` with `Output.object`, and the tRPC router for message persistence and retrieval.

## Requirements

- AI-02: 用户可通过对话方式修正和精炼 AI 生成的结构化模型
- AI-03: AI 自动识别需求中的隐含假设并标注置信度

## must_haves

- `/api/ai/converse` POST route accepts `{ messages, requirementId, currentModel }`, calls `streamText` with `Output.object({ schema: ConversationResponseSchema })`, uses `convertToModelMessages`, returns `result.toUIMessageStreamResponse()`
- tRPC `conversationRouter` has `getMessages` (paginated, last 20, cursor-based) and `saveMessage` procedures
- `conversationRouter` is registered in `src/server/trpc/root.ts` as `conversation`

## Context

### Existing patterns to follow

- `Output.object` with native Zod 4 schema — no `jsonSchema()` bridge needed (verified in Phase 2)
- Use `toUIMessageStreamResponse()` NOT `toTextStreamResponse()` — `useChat` expects the UI message stream protocol
- Always call `convertToModelMessages(messages)` before passing to `streamText` — `useChat` uses `UIMessage` format, `streamText` expects `CoreMessage`
- `verifySession()` from `@/lib/dal` for auth guard on API route
- `protectedProcedure` from `@/server/trpc/init` for tRPC procedures
- `prisma` singleton from `@/server/db/client`
- `eventBus` from `@/server/events/bus` — emit `conversation.message.saved` after saving a message
- Follow the pattern from `src/server/trpc/routers/requirement.ts` for the router structure
