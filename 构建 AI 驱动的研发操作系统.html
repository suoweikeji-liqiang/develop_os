1) 先定两条总原则（写进团队标准里）

原则A：业务逻辑必须可脱离框架运行（Framework-agnostic）

  • 业务规则、计算、状态机、权限判断、策略选择等，都放在纯函数/纯类里

  • HTTP、DB、MQ、缓存、第三方API 只是“适配器/外壳”

原则B：所有外部依赖必须可替换（Dependency Inversion）

  • 任何“不可控”的东西（时间、随机、网络、DB、文件、全局配置）都必须通过接口/注入进入业务层

  • 这样 LLM 才能“用 fake/mock 一把梭”稳定写测试

你要的“不可测不让入库”，本质就是：禁止把业务写在不可替换的外部依赖里面。

2) 用结构保证可测：分层 + 依赖方向

推荐一个非常适合“LLM写代码 + 写测试”的结构（前后端都适用）：

后端（以 API 服务为例）

  • domain：纯业务（实体、值对象、规则、策略）✅（最容易单测、最值得单测）

  • application：用例/服务编排（一个接口背后做哪几步）✅（单测/组件测）

  • adapters：DB/HTTP/MQ/第三方（实现接口）✅（集成测/契约测）

  • entrypoints：Controller/Router（只做输入输出映射）✅（少量冒烟）

关键门禁：

  • domain 不允许 import 任何 web/db 框架包

  • application 只能依赖 domain + ports(interface)

  • adapters 依赖 application 的 ports，并实现它

前端（以 Web SPA 为例）

  • ui：组件渲染（尽量薄）

  • state/actions：状态管理、事件处理（可测）

  • services：API 调用封装（可 mock）

  • domain：前端也可以有纯逻辑（表单规则、计算、权限、流程状态机）

关键门禁：

  • 组件里禁止直接写复杂业务逻辑（否则很难测、LLM也难写测）

  • 复杂逻辑下沉到 domain/actions，组件只管“显示 + 派发事件”

3) 测试分层：让 LLM “按模板自动补齐”

把测试分成 4 层，并规定每层的必备产物：

  1. 单元测试（Unit）：domain 纯逻辑（快、稳定、覆盖高）

  2. 组件/用例测试（Component / Application）：application 的用例流程（mock DB/第三方）

  3. 契约测试（Contract）：前后端接口 schema/响应约定（防止“接口漂移”）

  4. 端到端（E2E）：关键用户路径 3~10 条（少而精）

硬标准建议：

  • domain 必须有单测（覆盖核心规则）

  • 每个 API endpoint 必须有契约测试（至少 schema）

  • 每个用户关键流程必须有1条 E2E（比如登录→进入主页→创建→查询→删除）

  • 断言点不清晰（业务规则没抽象）
